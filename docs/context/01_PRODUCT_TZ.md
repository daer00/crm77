# CRM77 — Единый контекст проекта

## Цель
Сделать CRM-систему с веб-интерфейсом (desktop + mobile), где пользователи работают с контактами и сделками, а администратор управляет доступами и подтверждением удаления.

## Источник данных
- Основной входной файл: `main_data.csv`.
- Fallback-файл: `CONTACT_20250916_5299b658_68c9c36edbc7d.csv`.
- Импорт в БД делается через backend-слой (не напрямую во фронт).

## Технические решения (подтверждено)
- Основная БД: PostgreSQL.
- SMTP-провайдер для кода входа: Yandex SMTP.
- Роли: `admin` и `user` с похожим UI (единая дизайн-система, разные права).
- Визуальный стиль: как в референсе (светлый верхний бар + акцентные синие/темно-синие hero-блоки).

## Архитектурные слои
- **БД**: контакты, сделки, файлы сделок.
- **Backend**: CRUD, фильтры, поиск, дедупликация, авторизация, роли, согласование удалений.
- **Frontend**: ЛК пользователя + ЛК администратора, адаптивность под мобильные устройства.

## Модели БД (целевая версия)
### Контакты
`IDc; Имя; Фамилия; Компания; Должность; Тип контакта; Рабочий телефон; Мобильный телефон; Рабочий e-mail; Личный e-mail; Источник; Область интереса; Дата добавления; Дата обновления`

### Сделки
`IDd; Название сделки; Компания; Дата начала сделки; Статус сделки; Область сделки; Стоимость сделки; Себестоимость сделки; Маржинальность сделки; Контакт для сделки 1; Контакт для сделки 2; Контакт для сделки 3; Контакт для сделки 4; Файл сделки 1; Файл сделки 2; Файл сделки 3; Файл сделки 4; Файл сделки 5; Дата окончания сделки`

### Файлы сделок
`IDf; IDd; Название сделки; файл`

## ТЗ на пользовательскую часть
1. Главная:
   - компании по группам, сортировка по убыванию;
   - быстрый вход в контакты компании;
   - добавить контакт и поиск контактов;
   - блок источников с переходом к списку контактов по источнику;
   - добавить сделку и поиск по сделкам.
2. Страница контактов:
   - пагинация по 50;
   - фильтры + сортировки;
   - создание контакта;
   - карточка отдельного контакта.
3. Страница сделок:
   - пагинация по 50;
   - фильтры + сортировки;
   - создание сделки;
   - карточка отдельной сделки.

## ТЗ на админскую часть
- Страница пользователей (создание аккаунтов).
- Страница подтверждения удаления.
- Удаление контактов/сделок только после подтверждения администратором.

## Авторизация
- Логин пользователя.
- Код подтверждения отправляется на email.
- Безопасные сессии и ролевой доступ (admin/user).

## Ограничения
- Никогда не читать и не изменять `.env` через агента.
- Агент может только указывать, какие переменные добавить в `.env`.
